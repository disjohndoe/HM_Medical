# 2.2 Ponavljajući (hr-create-health-issue-recurrence-message) rejected with ERR_HEALTH_ISSUE_2000

## Discovery

Wired up end-to-end 2.2 flow (FE "Ponovi slučaj" action on resolved cases →
backend dispatch → Certilia signing → CEZIH POST). CEZIH signed the bundle
successfully but the state machine rejected the POST with:

```
HTTP 400
{
  "code": "ERR_HEALTH_ISSUE_2000",
  "display": "Unexpected error occurred while executing state machine perform.",
  "diagnostics": "FHIR error: FHIR issue code: not-supported"
}
```

Tested on prod (app.hmdigital.hr) via Certilia against parent case:
- Parent: Zatvoren J20 "Akutni bronhitis" (case_id `cmo5rif7001wthb85f3i74hiq`)
- Patient: GORAN PACPRIVATNICI19 (MBO 999990260)

Bundle matches the documented profile contract in

> Profile: hr-create-health-issue-recurrence-message|0.1

Identifier IS stripped after build (`condition.pop("identifier", None)`).
Extsigner signing succeeds (same JWS format that 2.1/2.4/2.9 use). Bundle
validates locally.

Pre-flight GET `Condition?_count=0` on `health-issue-services` returns 400
`Missing mandatory parameter "patient"` (documented non-fatal warning, same
as for all 2.x paths). Session cookie warms and the POST proceeds.

## Impact

- TC17b Ponavljajući (2.2) — **cannot ship**. State machine rejects signed
  bundle. Not a blocker for certification (2.2 is not in the 22 TCs), but
  is a documented feature gap.
- Backend code path (FE → dispatch → service → builder → FHIR client) is
  otherwise healthy. Fix is likely narrow — a missing or mis-shaped field
  in the `hr-create-health-issue-recurrence-message` profile.
- FE action "Ponovi slučaj" was deployed briefly and reverted in the same
  commit chain. Left the backend dispatcher local-DB parent lookup
  (replaces flaky `real_service.retrieve_cases` qedm call) since that's a
  real improvement.

## Hypotheses to investigate (not yet tried)

1. **MessageHeader.eventCoding.system mismatch** — bundle uses
   `http://ent.hr/fhir/CodeSystem/ehe-message-types`, but other CEZIH-verified
   events (2.1/2.4/2.9) also use this same system and are accepted. So
   probably not this.
2. **Missing `Condition.clinicalStatus`** — 2.2 profile may want
   `clinicalStatus=recurrence` (analogous to 2.4 requiring `resolved`).
   Profile doc says only "identifier forbidden, ICD+verification+onset
   required"; clinicalStatus may be implicitly required.
3. **Missing link to parent Condition** — a recurrence that CEZIH can't
   trace to a prior resolved Condition may get rejected. `Condition.evidence`
   or `Condition.extension` with parent case identifier may be needed.
4. **`asserter` vs `recorder`** — profile may slice differently for
   recurrences vs new cases.
5. **`encounter` reference required** — per `case-lifecycle-profile-matrix.md`
   some profiles want an active Encounter at the time of the message.
