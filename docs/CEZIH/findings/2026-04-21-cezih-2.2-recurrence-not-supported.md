# 2.2 Ponavljajući (hr-create-health-issue-recurrence-message) — RESOLVED (H2a: keep local identifier)

## Resolution — 2026-04-21

**Root cause:** `create_recurring_case` was stripping the entire
`Condition.identifier` array (`condition.pop("identifier", None)`) under
the incorrect assumption "identifier FORBIDDEN". The 2.2 profile only
forbids the `globalni-identifikator` slice (`max=0`); the
`lokalni-identifikator` slice inherits from `hr-condition` base with
`max=*` and is valid. Sending a Condition with zero identifiers caused
the CEZIH state machine to reject with `ERR_HEALTH_ISSUE_2000`.

**Fix:** commit `90ab916` — replace blanket pop with a filter that
removes only entries whose `system` equals `ID_CASE_GLOBAL`, preserving
the local-identifier entry the builder creates.

**Verification (live, real CEZIH test env):**
- Parent: Zatvoren J20 `cmo5rif7001wthb85f3i74hiq`, patient GORAN
  PACPRIVATNICI19 (MBO 999990260)
- Extsigner (Certilia) signed the bundle successfully
- CEZIH `$process-message`: **HTTP 200 in 1378.4ms**, no ERR_HEALTH_ISSUE_2000
- Bundle verified in backend log — Condition carries exactly one
  identifier with `system = http://fhir.cezih.hr/specifikacije/identifikatori/lokalni-identifikator-slucaja`
- Backend request log: `PUT /api/cezih/cases/.../status → 200 (27443.6ms)`
  (total incl. Certilia signing)

**Prior hypothesis (H1 mirror — drop asserter):** FAILED. See H1 section
below. Asserter was NOT the 2.2 root cause, unlike 2.6. The two events
diverged at the state-machine validation layer; H1 solved 2.6 and H2a
solved 2.2.

**Known follow-up gap (not a CEZIH issue):**
CEZIH accepted the recurrence and assigned a new global case identifier
in the response Bundle, but our `parse_message_response` /
`dispatch_update_case` did not extract + persist a new row in
`cezih_cases`. FE response was `{"success":true,"case_id":null,"action":null}`.
So the on-CEZIH side is correct; our mirror is stale. Separate fix:
inspect the 2.2 response shape and update the parser to find the new
`identifier:globalni-identifikator` on the returned Condition. Track in
next session — does NOT block TC17 cert scope because the cert exam
verifies the wire-level bundle is accepted, not local persistence.

**Conclusion:** asserter is NOT the 2.2 root cause. 2.6 fix was

## Hypotheses to investigate (not yet tried)

1. **MessageHeader.eventCoding.system mismatch** — bundle uses
   `http://ent.hr/fhir/CodeSystem/ehe-message-types`, but other CEZIH-verified
   events (2.1/2.4/2.9) also use this same system and are accepted. So
   probably not this.
2. **Missing `Condition.clinicalStatus`** — 2.2 profile may want
   `clinicalStatus=recurrence` (analogous to 2.4 requiring `resolved`).
   Profile doc says only "identifier forbidden, ICD+verification+onset
   required"; clinicalStatus may be implicitly required.
3. **Missing link to parent Condition** — a recurrence that CEZIH can't
   trace to a prior resolved Condition may get rejected. `Condition.evidence`
   or `Condition.extension` with parent case identifier may be needed.
4. **`asserter` vs `recorder`** — profile may slice differently for
   recurrences vs new cases.
5. **`encounter` reference required** — per `case-lifecycle-profile-matrix.md`
   some profiles want an active Encounter at the time of the message.
