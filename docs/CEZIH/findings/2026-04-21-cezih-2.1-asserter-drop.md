# 2.1 Create case rejected with ERR_HEALTH_ISSUE_2000 — H1 mirror (drop asserter)

## Discovery

TC16 Create Case via Certilia mobile on Croatian patient GORAN PACPRIVATNICI19
(MBO 999990260) silently reset form. Backend logs show extsigner round-trip
succeeded (NOTIFICATION_SENT → signed bundle retrieved) but CEZIH
`$process-message` on `health-issue-services` returned HTTP 400 with the same
state-machine error that previously broke 2.2 and 2.6:

```
OperationOutcome.issue[0].severity=error
OperationOutcome.issue[0].code=not-supported
details.coding = ERR_HEALTH_ISSUE_2000
display = "Unexpected error occurred while executing state machine perform."
```

## Root cause

`build_condition_create` (in `backend/app/services/cezih/builders/condition.py`)
always emits `Condition.asserter`. Prior H1 fixes dropped asserter from 2.2
(`create_recurring_case`, commit 6b29da4) and 2.6 (`build_condition_data_update`,
commit 5cb984c), but **2.1 `create_case` never got the mirror** — it still
shipped asserter on the wire.

Working lifecycle operations (2.4 Resolve, 2.9 Reopen) never emit asserter;
the CEZIH state machine is stricter than the Simplifier profile
(`cezih.hr.condition-management/0.2.1`), which allows asserter max=1 mustSupport.

Evidence that TC16 previously passed (Phase 18, 2026-04-13) suggests CEZIH's
state machine was tightened between 2026-04-13 and 2026-04-21 — same window as
the 2.2/2.6 regressions.

## Fix

`backend/app/services/cezih/fhir_api/condition.py` — `create_case` pops
`asserter` after the builder returns, mirroring the 2.2 pattern:

```python
condition.pop("asserter", None)
```

`practitioner_id` still flows into `MessageHeader.author` via
`build_message_bundle`, so provenance is preserved at the envelope level.
