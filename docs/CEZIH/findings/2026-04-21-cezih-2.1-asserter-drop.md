# SUPERSEDED — 2.1 ERR_HEALTH_ISSUE_2000 was CEZIH transient, not a payload bug

**This finding's root-cause analysis was wrong.** The H1 asserter drop from 2.2/2.6
does NOT apply to 2.1. TC16 resolved by retry on 2026-04-21 — see
`2026-04-21-cezih-2.1-transient-state-machine-err.md`.

## What actually happened

TC16 Create Case rejected repeatedly with `ERR_HEALTH_ISSUE_2000`
("Unexpected error occurred while executing state machine perform").
Guessed that CEZIH had tightened the state machine to reject `Condition.asserter`
on event code 2.1 — same pattern as the 2.2/2.6 regressions from earlier H1.

Three hypotheses attempted in order, all failed the same way:

1. **Drop asserter** (commit 544e31a) — still `ERR_HEALTH_ISSUE_2000`
2. **Add `meta.profile` annotations** for `hr-create-health-issue-message` +
   `hr-hi-management-message-header` — still `ERR_HEALTH_ISSUE_2000`
3. **Restore asserter** (commit a920ed9, kept meta.profile) — still
   `ERR_HEALTH_ISSUE_2000`
4. **Change verificationStatus to `unconfirmed`** — still `ERR_HEALTH_ISSUE_2000`

After exhausting payload hypotheses, retried the SAME bundle a minute later
→ **HTTP 200, case `cmo8bxbcu023ehb85nzsggu23` created**. Identical payload,
different outcome — confirms CEZIH test-env transient, not a payload defect.

## Cleanup state (as of 2026-04-21 09:54 CET)

- `backend/app/services/cezih/fhir_api/condition.py` has `asserter` RESTORED on
  2.1 (matches 2026-04-16 verified shape).
- `meta.profile` URLs added to 2.1 bundle + MessageHeader remain — harmless,
  leave as-is until further study.
- No further code changes needed to unblock TC16.

## Lessons

`ERR_HEALTH_ISSUE_2000` joins `ERR_DOCTRANSVAL_1000` as a known CEZIH-side
transient flake class. Before diagnosing either as a payload bug, **retry
the exact same request within ~60s**. Only if the retry also fails is it
worth chasing payload changes.

See `2026-04-21-cezih-2.1-transient-state-machine-err.md` for the consolidated
finding.
